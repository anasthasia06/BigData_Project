# 🧠 **Synthèse globale : Architecture des grandes plateformes data (Netflix, Uber, Airbnb, Steam)**

---

## **1. Types de problèmes selon les plateformes**

Chaque plateforme est dominée par un type de charge spécifique :

| Plateforme  | Problème principal                       | Nature des données                         |
| ----------- | ---------------------------------------- | ------------------------------------------ |
| **Netflix** | Écritures massives (historique)          | Logs linéaires, événements utilisateurs    |
| **Uber**    | Événements temps réel                    | Données géospatiales, très volatiles       |
| **Airbnb**  | Matching + rareté                        | Objets complexes, inventaire unique        |
| **Steam**   | Lecture massive + transactions critiques | Inventaires numériques, économie virtuelle |

👉 **Conclusion clé :**
Il n’existe pas une seule architecture optimale → tout dépend du type de données et du problème métier.

---

## **2. Pattern commun : séparation des systèmes**

Toutes ces plateformes utilisent une séparation fondamentale :

### 🔹 **OLTP (Opérationnel)**

* Temps réel
* Faible latence
* Actions critiques (achat, matching, lecture immédiate)

### 🔹 **OLAP (Analytique)**

* Batch / différé
* Machine Learning
* Optimisation (pricing, recommandation, fraude)

👉 Elles sont reliées par des **pipelines d’événements (Kafka-like)**.

---

## **3. Event-driven architecture (le cœur moderne)**

Toutes les plateformes reposent sur une logique commune :

👉 **Tout est un événement**

Exemples :

* Netflix → WATCH_START, WATCH_COMPLETE
* Uber → GPS_UPDATE, TRIP_STATE
* Airbnb → BOOKING_CONFIRMED
* Steam → PURCHASE, ITEM_GRANTED

### ✔️ Avantages :

* audit complet
* reconstruction de l’état
* scalabilité
* découplage des services

👉 Ce pattern est appelé **Event Sourcing**

---

## **4. CQRS : séparation lecture / écriture**

👉 Toutes les plateformes séparent :

* **Command side (write)** → optimisé pour écrire
* **Query side (read)** → optimisé pour lire

➡️ appelé **CQRS**

### Pourquoi ?

* Les writes et reads ont des contraintes opposées
* Permet d’optimiser indépendamment
* Supporte le scaling massif

---

## **5. Polyglot persistence (pas une seule base)**

👉 Aucune plateforme n’utilise une seule base de données.

Elles combinent :

* SQL → transactions (paiement, inventaire)
* NoSQL → volume et vitesse
* Cache (Redis) → temps réel
* Search (Elasticsearch) → recherche
* Data lake (S3/HDFS) → historique

➡️ appelé **Polyglot Persistence**

---

## **6. Gestion des compromis (trade-offs)**

Chaque plateforme optimise un axe au détriment d’un autre :

| Problème | Choix                  | Compromis               |
| -------- | ---------------------- | ----------------------- |
| Netflix  | Écriture massive       | Complexité stockage     |
| Uber     | Latence temps réel     | Perte de vision globale |
| Airbnb   | Précision matching     | Complexité ML           |
| Steam    | Cohérence transactions | Latence + complexité    |

👉 Principe fondamental :
➡️ **On ne peut pas optimiser simultanément cohérence, latence et scalabilité** (CAP theorem implicite)

---

## **7. Rôle clé du Machine Learning**

Toutes les plateformes utilisent le ML pour :

* recommandations (Netflix, Steam, Airbnb)
* pricing dynamique (Uber, Airbnb)
* détection de fraude (Steam, Uber)
* ranking et matching

👉 Le ML est toujours alimenté par **les données collectées via les événements**.

---

## **8. Vision unifiée (modèle mental)**

👉 Toutes ces architectures suivent le même schéma global :

```
Users → Events → Event Bus → 
    → OLTP (temps réel)
    → OLAP (analyse / ML)
```

Puis :

* OLTP → réponse immédiate
* OLAP → amélioration du système

---

# 🎯 **Conclusion finale**

👉 Ces plateformes ne sont pas différentes par la technologie…
👉 Elles sont différentes par **leur problème métier dominant**.

Mais elles convergent toutes vers :

* architecture orientée événements
* séparation read/write (CQRS)
* multi-bases (polyglot)
* pipelines data massifs
* ML intégré

# 🎮 **Synthèse – Choix technologiques et architecture de Steam**

---

## **1. Positionnement du problème**

Steam fait face à un **problème hybride unique** :

* 🔹 **Lecture massive dominante**
  (catalogue, pages jeux, reviews, profils)

* 🔹 **Écritures rares mais critiques**
  (achats, inventaires, marketplace)

* 🔹 **Temps réel**
  (amis en ligne, matchmaking, sessions)

👉 Contrairement à Netflix ou Uber, Steam doit **combiner plusieurs contraintes fortes simultanément**.

---

## **2. Choix clé n°1 : Séparer les systèmes selon leur rôle**

Steam adopte une architecture découpée en couches :

* **OLTP (transactions critiques)**
  → achats, inventaires, marketplace
  → cohérence forte (ACID)

* **Stores temps réel (in-memory)**
  → présence, sessions
  → latence ultra faible

* **Systèmes lecture optimisés**
  → catalogue, recherche
  → caches massifs + CDN

👉 Chaque composant est optimisé pour **un seul type de besoin**.

---

## **3. Choix clé n°2 : Priorité à la cohérence sur les opérations sensibles**

Steam traite :

* argent réel (achats)
* biens numériques (skins, items)

➡️ donc :

* aucune perte de données tolérée
* audit complet nécessaire
* transactions strictement contrôlées

👉 Steam privilégie **la fiabilité plutôt que la vitesse** sur ces flux.

---

## **4. Choix clé n°3 : Architecture orientée événements**

Toutes les actions importantes sont stockées comme événements :

* PURCHASE
* ITEM_GRANTED
* TRADE
* GAME_START

👉 Cela correspond au pattern :
→ Event Sourcing

### Objectifs :

* audit anti-fraude
* reconstruction d’inventaire
* traçabilité complète

---

## **5. Choix clé n°4 : Séparation lecture / écriture**

Steam applique une logique proche de :

→ CQRS

* **Writes (command side)**
  → transactions, inventaire
  → cohérence forte

* **Reads (query side)**
  → catalogue, profils
  → optimisé pour la vitesse

👉 Cela permet de gérer la **charge massive de lecture sans impacter les écritures critiques**.

---

## **6. Choix clé n°5 : Polyglot persistence**

Steam utilise plusieurs types de bases :

* SQL → transactions
* NoSQL → données volumineuses
* Redis-like → temps réel
* Elasticsearch → recherche
* Object storage → fichiers jeux / cloud saves

👉 Pattern :
→ Polyglot Persistence

---

## **7. Choix clé n°6 : CDN massif pour la distribution**

Steam possède un des plus grands CDN privés :

* distribution des jeux
* mises à jour
* contenus Workshop

👉 Permet :

* réduction de la latence mondiale
* scalabilité énorme côté téléchargement

---

## **8. Trade-off principal**

Steam optimise :

➡️ **Cohérence + scalabilité lecture**

au prix de :

* complexité architecturale élevée
* systèmes multiples à maintenir
* orchestration difficile

---

# 🎯 **Conclusion**

👉 Steam n’est pas une simple plateforme de jeux :
c’est une combinaison de :

* marketplace
* système financier
* réseau social temps réel
* CDN global
* moteur de recommandation

👉 Son choix architectural repose sur une idée clé :

> **séparer chaque problème (lecture, écriture, temps réel, distribution)
> et utiliser le meilleur système pour chacun.**
